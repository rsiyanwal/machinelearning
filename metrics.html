Teaching Overview

How to use this tutorial

This page is organized the way you would teach it in class: start with the confusion matrix, then move metric-by-metric, then end with a decision framework. Every important metric is attached to a concrete dataset so students can see why that metric exists.

Core principle: no metric is universally best. The right metric depends on class imbalance and on which mistake costs more in the real world.
At a Glance
Accuracy

Useful when classes are balanced and both error types have similar cost.

Precision

Useful when false positives are costly.

Recall

Useful when false negatives are costly.

F1 / Weighted F1

Useful when classes are imbalanced and you need balanced or class-frequency-aware reporting.

P0

Part 0 — The Confusion Matrix (Foundation)

Before any metric makes sense, you need to understand what the model got right and what it got wrong.

Before any metric makes sense, you need to understand what a model is actually doing when it classifies something. It is comparing its prediction against ground truth (the real answer). Every prediction lands in one of four buckets.

The Four Outcomes

Imagine a binary classifier that predicts Spam (Positive) or Not Spam (Negative).

PREDICTED Positive Negative ┌───────────────┬───────────────┐ Positive │ TP │ FN │ ACTUAL │ True Positive │ False Negative│ ├───────────────┼───────────────┤ Negative │ FP │ TN │ │ False Positive │ True Negative│ └───────────────┴───────────────┘
Cell Full Name What it means
TPTrue PositiveModel said Positive. It was right.
TNTrue NegativeModel said Negative. It was right.
FPFalse PositiveModel said Positive. It was wrong. (Type I Error)
FNFalse NegativeModel said Negative. It was wrong. (Type II Error)

Dataset 0 — Email Spam Filter

A company deploys a spam filter. Over one week, 200 emails arrive: 90 are spam, 110 are legitimate.

Actual Spam emails: 90 Actual Legit emails: 110 Total: 200

The model produces this confusion matrix:

PREDICTED Spam Not Spam ┌───────────────┬───────────────┐ Spam │ TP = 78 │ FN = 12 │ (90 actual spam) ACTUAL │ │ │ ├───────────────┼───────────────┤ Not Spam │ FP = 6 │ TN = 104 │ (110 actual legit) └───────────────┴───────────────┘ (84 predicted spam) (116 predicted not spam)

Reading this:

  • 78 spam emails were correctly caught (TP)
  • 12 spam emails slipped through to the inbox (FN) — missed
  • 6 legitimate emails were wrongly sent to spam (FP) — collateral damage
  • 104 legitimate emails correctly reached the inbox (TN)
Teaching takeaway: this single matrix is the raw material for every metric below.
P1

Part 1 — Type I and Type II Errors

False alarms versus missed detections.

These come from statistics, but they are fundamental to understanding what can go wrong with a classifier.

The Statistical Frame

In hypothesis testing, you start with a null hypothesis (H₀). In classification, the null hypothesis is always: "This instance is Negative."

Error Type Also Called What happened
Type IFalse PositiveYou rejected H₀ when H₀ was true. You said positive, but the truth was negative.
Type IIFalse NegativeYou failed to reject H₀ when H₀ was false. You said negative, but the truth was positive.

A simple way to remember:

  • Type I → False Alarm
  • Type II → Missed Detection

Dataset 1A — COVID Screening Test

A rapid COVID test is evaluated on 2,000 patients. 200 actually have COVID; 1,800 are healthy.

H₀ = "Patient does NOT have COVID" (they are healthy)
TEST RESULT Positive Negative ┌───────────────┬───────────────┐ COVID │ TP = 181 │ FN = 19 │ 200 actual cases ACTUAL │ │ │ ├───────────────┼───────────────┤ Healthy │ FP = 54 │ TN = 1746 │ 1800 healthy └───────────────┴───────────────┘

Type I Errors (FP = 54): 54 healthy people tested positive. They experienced unnecessary anxiety, isolation, and follow-up PCR tests. Cost: psychological distress and economic disruption.

Type II Errors (FN = 19): 19 infected people tested negative. They were not isolated and may have unknowingly spread the virus. Cost: potential outbreak.

Dataset 1B — Airport Security Screening

Now consider airport security. A scanner flags passengers as either "clear" or "threat-detected."

H₀ = "Passenger is NOT carrying a weapon"
SCANNER RESULT Flag Clear ┌───────────────┬───────────────┐ Threat │ TP = 8 │ FN = 2 │ 10 actual threats ACTUAL │ │ │ ├───────────────┼───────────────┤ Clear │ FP = 950 │ TN = 99040 │ 99990 safe passengers └───────────────┴───────────────┘

Type I Errors (FP = 950): Nearly 1,000 innocent passengers are flagged and delayed. Operationally costly and creates poor experience, but not catastrophic.

Type II Errors (FN = 2): 2 actual threats pass through undetected. Potentially catastrophic.

The Core Trade-off

These two scenarios reveal something critical: the cost of Type I vs Type II errors depends entirely on the domain.

Domain Worse Error Why
Cancer detectionType II (FN)Missing cancer means no treatment, patient may die
Spam filterType I (FP)Losing a legitimate email is worse than getting spam
Loan approvalType I (FP)Approving a bad borrower costs money
Airport securityType II (FN)Missing a weapon can be catastrophic
Drug trialType I (FP)Approving ineffective drug wastes resources and harms
Fraud detectionType II (FN)Missed fraud means financial loss

There is always a trade-off. Reducing one type of error generally increases the other. This is why no single metric tells the full story.

P2

Part 2 — Accuracy

Useful when the setup is balanced. Dangerous when it is not.

The Formula

Accuracy = (TP + TN) / (TP + FP + FN + TN) = Correct predictions / All predictions

Accuracy answers: "What fraction of all my predictions were correct?"

Dataset 2A — When Accuracy Works: Balanced Sentiment Classifier

A product review classifier labels reviews as Positive or Negative. The dataset is intentionally balanced.

Total reviews: 1,000 Actual Positive: 500 Actual Negative: 500
PREDICTED Positive Negative ┌───────────────┬───────────────┐ Positive │ TP = 455 │ FN = 45 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Negative │ FP = 38 │ TN = 462 │ └───────────────┴───────────────┘
Accuracy = (455 + 462) / 1000 = 917 / 1000 = 91.7%

This is a fair, meaningful number. The classes are balanced. If the model is wrong, it is equally wrong on both classes. Accuracy is a perfectly reasonable metric here.

Dataset 2B — When Accuracy Lies: Rare Disease Screening

A hospital screens 10,000 patients for a rare disease. The disease affects 1% of the population.

Total patients: 10,000 Actual positive: 100 (1%) Actual negative: 9,900 (99%)

Model A — A lazy model that predicts "Healthy" for everyone:

PREDICTED Disease Healthy ┌───────────────┬───────────────┐ Disease │ TP = 0 │ FN = 100 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Healthy │ FP = 0 │ TN = 9,900 │ └───────────────┴───────────────┘
Accuracy = (0 + 9900) / 10000 = 99.0%

99% accuracy. But the model never detected a single case of the disease. It is completely useless for its purpose. Every sick patient was sent home undiagnosed.

Model B — A real, trained classifier:

PREDICTED Disease Healthy ┌───────────────┬───────────────┐ Disease │ TP = 91 │ FN = 9 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Healthy │ FP = 120 │ TN = 9,780 │ └───────────────┴───────────────┘
Accuracy = (91 + 9780) / 10000 = 9871 / 10000 = 98.7%

Model B has lower accuracy (98.7%) than the lazy model (99%), yet it correctly detected 91 out of 100 sick patients and is vastly more useful.

The Lesson from Accuracy

  • Useful when classes are approximately balanced
  • Misleading when there is class imbalance
  • Never sufficient alone in medical, fraud, or safety applications
When you see a model with 99% accuracy and the positive class is rare, ask: "What would happen if this model predicted the majority class for everything?"
P3

Part 3 — Precision

When you raise the alarm, how often are you right?

The Formula

Precision = TP / (TP + FP)

Precision answers: "Of all the times I predicted Positive, how often was I actually right?"

It only cares about the positive predictions the model made. FN and TN do not appear in the formula. They are irrelevant to this question.

Dataset 3 — Content Recommendation (Reducing False Alarms)

A streaming platform builds a recommendation engine to surface "movies the user will love" (Positive) vs "movies the user will not like" (Negative). Over 500 recommendation slots, here is what 100 users reported.

The company cares about precision because showing a bad recommendation wastes screen real estate and erodes user trust. They can afford to miss some good movies — there are hundreds more — but they cannot afford to constantly show movies the user dislikes.

Model A — Aggressive (low threshold, recommends broadly):

PREDICTED Loved Not Loved ┌───────────────┬───────────────┐ Loved │ TP = 210 │ FN = 140 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Not Loved│ FP = 160 │ TN = - │ └───────────────┴───────────────┘ Precision = 210 / (210 + 160) = 210 / 370 = 56.8%

Barely better than random. Nearly half of all recommendations are movies the user dislikes.

Model B — Conservative (high threshold, only recommends confident picks):

PREDICTED Loved Not Loved ┌───────────────┬───────────────┐ Loved │ TP = 175 │ FN = 175 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Not Loved│ FP = 25 │ TN = - │ └───────────────┴───────────────┘ Precision = 175 / (175 + 25) = 175 / 200 = 87.5%

Model B misses many movies the user would have loved (high FN), but nearly every recommendation it makes is a good one. For this use case, that is the right trade-off.

Dataset 3B — Legal Document Review

A law firm uses a classifier to flag documents as "Relevant to case" or "Not relevant." Attorneys manually review every flagged document, which costs $400/hour. There are 50,000 documents.

Actual relevant: 800 (1.6%) Actual not relevant: 49,200 (98.4%)
PREDICTED Relevant Not Relevant ┌───────────────┬───────────────┐ Relevant │ TP = 760 │ FN = 40 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Not Relev. │ FP = 3,800 │ TN = 45,400 │ └───────────────┴───────────────┘ Precision = 760 / (760 + 3800) = 760 / 4560 = 16.7%

The attorneys must review 4,560 documents to find 760 relevant ones. Low precision = high wasted effort. Here, improving precision directly reduces cost.

The Lesson from Precision

Precision is the right metric when:

  • False positives are costly (wasted money, resources, trust)
  • You can afford to miss some positives
  • The downstream action triggered by a positive prediction is expensive or disruptive
Precision ignores false negatives entirely — which means a model can achieve perfect precision by only making one prediction and being very confident about it. Always pair it with recall.
P4

Part 4 — Recall (Sensitivity)

Of all the true positives that exist, how many did you find?

The Formula

Recall = TP / (TP + FN)

Recall answers: "Of all the actual Positives in the dataset, how many did I find?"

It only cares about the actual positive class. FP and TN do not appear in the formula. They are irrelevant to this question.

Also called Sensitivity or True Positive Rate.

Dataset 4A — Cancer Tumor Detection

A radiologist's assistant model scans mammograms and flags potential malignant tumors. Oncologists review all flagged cases. The dataset covers 1,000 scans: 100 are malignant, 900 are benign.

Missing a malignant tumor (FN) means the patient receives no treatment. The cancer progresses. The cost of a false negative is potentially a human life.

A false positive means the patient gets called back for a follow-up biopsy — stressful and inconvenient, but not dangerous.

Model A — Cautious (high precision, low recall):

PREDICTED Malignant Benign ┌───────────────┬───────────────┐ Malignant │ TP = 60 │ FN = 40 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Benign │ FP = 4 │ TN = 896 │ └───────────────┴───────────────┘ Precision = 60 / 64 = 93.8% Recall = 60 / 100 = 60.0%

Model A is very precise — when it flags something, it is almost always right. But it misses 40 malignant cases. Those 40 patients walk away with cancer undetected.

Model B — Sensitive (high recall, lower precision):

PREDICTED Malignant Benign ┌───────────────┬───────────────┐ Malignant │ TP = 97 │ FN = 3 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Benign │ FP = 148 │ TN = 752 │ └───────────────┴───────────────┘ Precision = 97 / 245 = 39.6% Recall = 97 / 100 = 97.0%

Model B catches 97 out of 100 malignant tumors. But it also flags 148 benign cases, sending those patients through unnecessary biopsies. The 3 missed cases are tragedies, but the 97 caught cases are lives saved.

For cancer detection, Model B is the clinically correct choice. High recall is mandatory. The false positive burden (biopsies) is acceptable because the false negative cost (untreated cancer) is unacceptable.

Dataset 4B — Flight Safety Alert System

An automated system monitors 500 flights for mechanical anomalies. There are 20 real faults. Every undetected fault risks catastrophic failure.

PREDICTED Fault No Fault ┌───────────────┬───────────────┐ Fault │ TP = 20 │ FN = 0 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ No Fault │ FP = 85 │ TN = 395 │ └───────────────┴───────────────┘ Recall = 20 / 20 = 100% Precision = 20 / 105 = 19.0%

100% recall. Not a single fault missed. 85 false alarms trigger unnecessary ground checks — costly, but every plane is safe. In aviation, perfect recall is the design goal.

The Lesson from Recall

Recall is the right metric when:

  • False negatives are catastrophic (missed diagnoses, undetected faults, missed fraud)
  • You can tolerate false alarms
  • The cost of missing a positive instance is much higher than investigating a false positive
Recall ignores false positives entirely — which means a model can achieve perfect recall by predicting Positive for everything. Always pair it with precision.
P5

Part 5 — F1 Score

When you need precision and recall to be good at the same time.

The Problem with Precision and Recall Alone

Given what we know:

  • A model that predicts Positive for everything achieves Recall = 100% but Precision collapses
  • A model that only makes one very confident Positive prediction achieves Precision = 100% but Recall collapses

We need a single number that balances both. That is the F1 Score.

The Formula

F1 = 2 × (Precision × Recall) / (Precision + Recall) = 2TP / (2TP + FP + FN)

The F1 Score is the harmonic mean of Precision and Recall. The harmonic mean punishes extreme imbalance between the two numbers far more severely than the arithmetic mean would.

Example: Precision = 90%, Recall = 10% Arithmetic mean = (0.90 + 0.10) / 2 = 50% ← looks decent Harmonic mean = 2(0.90)(0.10) / (0.90 + 0.10) = 18% ← reflects the problem

The harmonic mean forces both values to be high for F1 to be high.

Dataset 5 — Online Fraud Detection

A payment platform processes 100,000 transactions per day. 400 are fraudulent (0.4%). Three candidate models are evaluated.

Total transactions: 100,000 Actual fraud: 400 Actual legitimate: 99,600

Model A — The Precision Chaser:
Very conservative. Only flags near-certain fraud.

PREDICTED Fraud Legit ┌───────────────┬───────────────┐ Fraud │ TP = 280 │ FN = 120 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Legit │ FP = 14 │ TN = 99,586 │ └───────────────┴───────────────┘ Precision = 280 / 294 = 95.2% Recall = 280 / 400 = 70.0% F1 = 2(0.952)(0.70) / (0.952 + 0.70) = 80.7%

Very few legitimate transactions blocked. But 120 fraudulent transactions ($60,000 assuming $500 avg) go through daily.

Model B — The Recall Chaser:
Flags anything suspicious.

PREDICTED Fraud Legit ┌───────────────┬───────────────┐ Fraud │ TP = 384 │ FN = 16 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Legit │ FP = 2,400 │ TN = 97,200 │ └───────────────┴───────────────┘ Precision = 384 / 2784 = 13.8% Recall = 384 / 400 = 96.0% F1 = 2(0.138)(0.96) / (0.138 + 0.96) = 24.0%

Catches almost all fraud. But 2,400 legitimate customers have their transactions blocked daily. The customer service fallout alone would be ruinous. F1 = 24% correctly reflects this disaster.

Model C — Balanced:
Optimized for F1.

PREDICTED Fraud Legit ┌───────────────┬───────────────┐ Fraud │ TP = 348 │ FN = 52 │ ACTUAL │ │ │ ├───────────────┼───────────────┤ Legit │ FP = 120 │ TN = 99,480 │ └───────────────┴───────────────┘ Precision = 348 / 468 = 74.4% Recall = 348 / 400 = 87.0% F1 = 2(0.744)(0.87) / (0.744 + 0.87) = 80.1%

Side-by-side comparison

Model Precision Recall F1 Fraud missed/day Legit blocked/day
Model A95.2%70.0%80.7%12014
Model B13.8%96.0%24.0%162,400
Model C74.4%87.0%80.1%52120

Model A and C have similar F1 scores (80.7% vs 80.1%), but Model A has much better precision while C has better recall. The business decision about which to deploy depends on what costs more: fraud losses or customer churn from blocked legitimate transactions. The F1 score narrows the choice but does not eliminate business judgment.

The Lesson from F1

F1 is the right metric when:

  • Both precision and recall matter and you want a single number
  • Classes are imbalanced (more reliable than accuracy)
  • You need to compare multiple models quickly

F1 treats precision and recall as equally important. If your use case values one more than the other, consider the Fβ score:

Fβ = (1 + β²) × (Precision × Recall) / (β² × Precision + Recall)

When β > 1, recall is weighted more heavily (e.g., F2 for medical). When β < 1, precision is weighted more heavily (e.g., F0.5 for spam filters where false positives are expensive).

P6

Part 6 — Weighted F1

What to do when the task is multi-class and class sizes are uneven.

The Problem: Multi-class Imbalance

All the metrics above assumed binary classification. Real problems often have three or more classes — and those classes are rarely equally sized. A simple macro-average F1 (the unweighted average of per-class F1 scores) treats a class with 5 samples identically to a class with 5,000 samples.

Weighted F1 corrects for this by weighting each class's F1 score by its support — the number of actual instances of that class in the dataset.

Formula

For each class i:

F1ᵢ = 2 × Precisionᵢ × Recallᵢ / (Precisionᵢ + Recallᵢ)

Then:

Weighted F1 = Σ (support_i / total_samples) × F1_i = (support_1 × F1_1 + support_2 × F1_2 + ... + support_k × F1_k) / total_samples

Dataset 6 — Customer Support Ticket Classifier

A SaaS company auto-classifies incoming support tickets into three categories:

  • Billing — subscription, invoices, payments
  • Technical — bugs, downtime, integration issues
  • General — onboarding, feature requests, feedback

The distribution is heavily imbalanced because most customers hit technical issues.

Class | Support (actual count) -------------|------------------------ Billing | 180 tickets (18%) Technical | 650 tickets (65%) General | 170 tickets (17%) -------------|------------------------ Total | 1,000 tickets

The model produces these per-class confusion results (derived from the full 3×3 confusion matrix):

Per-class confusion matrix (simplified): Predicted Billing Predicted Technical Predicted General Billing 148 20 12 Technical 8 618 24 General 15 28 127

Step 1: Compute per-class metrics

Billing (class 0):

TP_billing = 148 FP_billing = 8 + 15 = 23 (other classes predicted as Billing) FN_billing = 20 + 12 = 32 (Billing predicted as something else) Precision_billing = 148 / (148 + 23) = 148 / 171 = 86.5% Recall_billing = 148 / (148 + 32) = 148 / 180 = 82.2% F1_billing = 2(0.865)(0.822) / (0.865 + 0.822) = 84.3% Support_billing = 180

Technical (class 1):

TP_tech = 618 FP_tech = 20 + 28 = 48 (other classes predicted as Technical) FN_tech = 8 + 24 = 32 (Technical predicted as something else) Precision_tech = 618 / (618 + 48) = 618 / 666 = 92.8% Recall_tech = 618 / (618 + 32) = 618 / 650 = 95.1% F1_tech = 2(0.928)(0.951) / (0.928 + 0.951) = 93.9% Support_tech = 650

General (class 2):

TP_general = 127 FP_general = 12 + 24 = 36 (other classes predicted as General) FN_general = 15 + 28 = 43 (General predicted as something else) Precision_general = 127 / (127 + 36) = 127 / 163 = 77.9% Recall_general = 127 / (127 + 43) = 127 / 170 = 74.7% F1_general = 2(0.779)(0.747) / (0.779 + 0.747) = 76.3% Support_general = 170

Step 2: Summary table

Class Precision Recall F1 Support
Billing86.5%82.2%84.3%180
Technical92.8%95.1%93.9%650
General77.9%74.7%76.3%170

Step 3: Compute macro F1 (unweighted)

Macro F1 = (84.3 + 93.9 + 76.3) / 3 = 254.5 / 3 = 84.8%

Macro F1 treats all classes equally, regardless of how many samples each has.

Step 4: Compute weighted F1

Weighted F1 = (180 × 84.3 + 650 × 93.9 + 170 × 76.3) / 1000 = (15,174 + 61,035 + 12,971) / 1000 = 89,180 / 1000 = 89.2%

Step 5: Understanding the difference

Macro F1 = 84.8% ← Treats all classes equally Weighted F1 = 89.2% ← Dominated by the Technical class (65% of data)

Weighted F1 is higher because the Technical class — which has the highest F1 (93.9%) — is also the largest class (65% of data). It dominates the weighted average.

This is important: if the model performs poorly on smaller classes (Billing, General), the weighted F1 will not punish it very harshly because those classes contribute less weight. Macro F1 would expose this more clearly.

Choosing Between Macro and Weighted F1

Metric Use when
Macro F1Every class is equally important regardless of size. (e.g., rare disease detection where the rare class matters as much as the common one)
Weighted F1Performance on larger classes matters more. (e.g., routing system where most tickets are of one type and accuracy on that type drives user satisfaction)
Micro F1Global TP/FP/FN counts matter. Equivalent to accuracy when classes are mutually exclusive. Dominated by large classes even more than Weighted F1.

What Happens in Extreme Imbalance

Consider this adversarial dataset for the same problem:

Class | Support ------------|-------- Class A | 950 (95%) Class B | 30 (3%) Class C | 20 (2%)

A model that achieves F1 = 95% on Class A, 0% on Class B, and 0% on Class C would produce:

Macro F1 = (95 + 0 + 0) / 3 = 31.7% ← Correctly signals failure Weighted F1 = (950×0.95 + 30×0 + 20×0) / 1000 = 90.25% ← Dangerously misleading

Weighted F1 of 90% would suggest the model is working well. Macro F1 of 32% immediately reveals the model is completely failing on two out of three classes. When minority classes matter, always report Macro F1 alongside Weighted F1.

Part 7 — Putting It All Together

A practical decision framework for metric selection.

Decision Framework: Which Metric to Use?

Is the task binary or multi-class? │ ├── Multi-class │ ├── Are all classes equally important? → Macro F1 │ ├── Do larger classes dominate importance? → Weighted F1 │ └── Is dataset highly imbalanced? → Report both + per-class breakdown │ └── Binary ├── Are classes balanced? │ └── Yes → Accuracy is fine. Add F1 for completeness. │ └── Classes imbalanced? ├── FN is catastrophic (cancer, fraud, safety) → Recall first, F1 second ├── FP is costly (spam, recommendations, legal review) → Precision first, F1 second └── Both matter roughly equally → F1 Score

Dataset 7 — Final Challenge: Choose the Right Metric

A financial institution builds three models for three different tasks using the same underlying technology. Same data pipeline, different objectives.

Task A — Mortgage Default Prediction (25,000 applicants, 2% default rate)

Should the bank use Accuracy, Precision, Recall, or F1?

The bank's cost of approving someone who defaults: $80,000 average loss. The bank's cost of rejecting someone creditworthy: $3,200 in lost lifetime revenue.

Since approving a bad borrower is far more costly than rejecting a good one, the model should be evaluated with a metric that strongly penalizes costly false approvals.

Teaching clarification: whether that bad approval is called a false positive or a false negative depends on how you define the positive class. If the positive class is "approve", it is a false positive. If the positive class is "will default", then missing the defaulter is a false negative. The business lesson stays the same: the high-cost mistake must dominate metric selection.

Since the high-cost mistake is much more expensive → Precision should be the primary metric if the model's positive class triggers approval, or more generally a metric focused on reducing bad approvals should dominate. Missing some good customers is acceptable. Approving bad borrowers is not.

Task B — Anti-Money Laundering Screening (1 million transactions, 0.01% suspicious)

Regulators mandate that suspicious activity reports are filed for flagged transactions. Missing an actual laundering operation (FN) triggers regulatory fines up to $50M. Investigating a false positive costs $200 per case.

Since FN cost >> FP cost, and the class is extremely rare → Recall as the primary metric, with F1 as secondary. Weighted accuracy would be completely misleading (99.99% by predicting nothing is suspicious).

Task C — Transaction Category Classifier (auto-labeling: Food, Transport, Entertainment, Utilities, Healthcare, Other)

Six classes, naturally imbalanced (Utilities = 4%, Healthcare = 3%, Other = 35%, etc.). The goal is to power a personal finance dashboard where users view breakdowns of all spending categories.

Here, all categories matter to the user. The Healthcare category has few transactions but high user sensitivity — mislabeling healthcare expenses is particularly frustrating.

→ Macro F1 as primary metric (all classes matter equally), supplemented with per-class F1 to monitor Healthcare and Utilities specifically. Report Weighted F1 as a secondary benchmark for overall system health.

Appendix — Quick Reference

All formulas and one-line intuition in one place.

All Formulas

Given: TP = True Positive, TN = True Negative, FP = False Positive, FN = False Negative Accuracy = (TP + TN) / (TP + TN + FP + FN) Precision = TP / (TP + FP) Recall = TP / (TP + FN) F1 = 2 × Precision × Recall / (Precision + Recall) = 2TP / (2TP + FP + FN) Fβ = (1 + β²) × Precision × Recall / (β² × Precision + Recall) Specificity = TN / (TN + FP) ← True Negative Rate FPR = FP / (FP + TN) ← False Positive Rate = 1 - Specificity FNR = FN / (FN + TP) ← Miss Rate = 1 - Recall For class i in a multi-class problem: Precision_i = TP_i / (TP_i + FP_i) Recall_i = TP_i / (TP_i + FN_i) F1_i = 2 × Precision_i × Recall_i / (Precision_i + Recall_i) Macro F1 = (1/K) × Σ F1_i where K = number of classes Weighted F1 = Σ (support_i / N) × F1_i where N = total samples Micro F1 = 2 × Σ TP_i / (2 × Σ TP_i + Σ FP_i + Σ FN_i)

Intuition in One Line Each

  • Accuracy: "How often am I right?"
  • Precision: "When I raise the alarm, is it real?"
  • Recall: "Do I catch everything I should?"
  • F1: "Am I both precise and thorough?"
  • Weighted F1: "Am I good across all classes, proportional to how common they are?"
  • Type I Error: "False alarm."
  • Type II Error: "Missed it."

The Never-Forget Mnemonic

Precision is about your predictions — of the positives you predicted, how many were real?

Recall is about reality — of the positives that actually exist, how many did you find?
